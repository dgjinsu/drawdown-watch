# Smart Commit - 변경점 분석 및 그룹별 커밋

코드 변경점을 분석하고, 논리적 그룹으로 묶어서 커밋합니다.

## 절차

### 1단계: 변경점 전체 파악

아래 명령어를 실행하여 현재 변경 사항을 파악하세요:

- `git status` 로 변경/추가/삭제된 파일 목록 확인
- `git diff` 로 수정된 파일의 상세 변경 내용 확인
- `git diff --cached` 로 이미 staged 된 변경 내용 확인
- `git log --oneline -5` 로 최근 커밋 스타일 확인

### 2단계: 변경점 분석 및 그룹핑

파악한 변경 사항을 아래 기준으로 논리적 그룹으로 분류하세요:

**그룹 분류 기준:**
- **기능(feat)**: 새로운 기능 추가
- **수정(fix)**: 버그 수정
- **리팩터(refactor)**: 기능 변경 없는 코드 개선
- **스타일(style)**: 코드 포맷팅, 세미콜론 등 (동작 변경 없음)
- **설정(chore)**: 빌드, 설정 파일, 의존성 변경
- **문서(docs)**: 문서 변경
- **테스트(test)**: 테스트 추가/수정

**그룹핑 원칙:**
- 같은 기능/목적에 해당하는 파일들은 하나의 그룹으로 묶기
- 서로 독립적인 변경은 별도 그룹으로 분리
- 하나의 그룹은 하나의 커밋이 되어야 함

### 3단계: 사용자 확인

분석 결과를 아래 형식으로 정리하여 사용자에게 보여주세요:

```
## 변경점 분석 결과

### 그룹 1: feat: [설명]
- 파일1 (변경 요약)
- 파일2 (변경 요약)

### 그룹 2: fix: [설명]
- 파일3 (변경 요약)

### 그룹 3: chore: [설명]
- 파일4 (변경 요약)
```

그룹핑이 적절한지, 커밋 메시지가 맞는지 사용자에게 확인을 받으세요.
AskUserQuestion 도구를 사용하여 "이대로 커밋을 진행할까요?" 라고 물어보세요.

### 4단계: 그룹별 순차 커밋

사용자가 승인하면, 각 그룹별로 순서대로:

1. 해당 그룹의 파일만 `git add <파일들>` 으로 staging
2. `git commit -m "<type>: <설명>"` 으로 커밋 (Conventional Commits 형식)
3. 다음 그룹으로 이동

**커밋 메시지 규칙:**
- 한글로 작성
- Conventional Commits 형식 사용: `feat:`, `fix:`, `refactor:`, `chore:`, `style:`, `docs:`, `test:`
- 50자 이내로 간결하게 작성
- 필요 시 본문에 상세 설명 추가

### 5단계: 결과 보고

모든 커밋 완료 후 `git log --oneline` 으로 결과를 보여주세요.
