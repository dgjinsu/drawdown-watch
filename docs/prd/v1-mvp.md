# PRD: MDD Watch MVP (v1)

## 1. 개요

- **한 줄 요약**: 주식 MDD(Maximum Drawdown) 임계값 초과 시 Telegram/Slack으로 알림하는 모니터링 서비스
- **작성일**: 2026-02-28
- **상태**: Draft
- **버전**: v1 MVP

## 2. 배경 및 목적

### 문제

- Portfolio Visualizer 같은 기존 분석 도구는 과거 데이터 백테스팅에 집중하며, 실시간 모니터링이나 알림 기능을 제공하지 않는다.
- 개인 투자자는 보유 종목의 MDD가 임계값을 넘는 시점을 인지하지 못해 손실을 방치하는 경우가 많다.
- 매일 차트를 직접 확인하는 것은 직장인 투자자에게 현실적으로 어렵다.

### 해결

- MDD Watch는 등록한 종목의 MDD를 매일 자동 계산하고, 사용자가 설정한 임계값을 초과하면 Telegram 또는 Slack으로 즉시 알려준다.
- "모니터링 자동화"를 통해 투자자가 수동으로 매일 차트를 확인하는 부담을 제거한다.

### 차별화

| 관점 | Portfolio Visualizer | MDD Watch |
|------|---------------------|-----------|
| 핵심 용도 | 과거 데이터 백테스팅 | 실시간 MDD 모니터링 + 알림 |
| MDD 분석 | 백테스트 결과의 일부 | MDD 자체가 핵심 기능 |
| 알림 | 없음 | Telegram/Slack 즉시 알림 |
| 실시간성 | 과거 데이터 분석만 | 매일 장 마감 후 자동 갱신 |

## 4. 핵심 결정 사항

> 기술 구현 방법이 아닌, "무엇을 기준으로 삼을 것인가"의 비즈니스 결정.

| 항목 | 결정 | 근거 |
|------|------|------|
| 대상 시장 | 한국(KRX) + 미국(NASDAQ/NYSE) | 국내 투자자의 해외 주식 비중 증가 |
| 시세 데이터 | 외부 API를 통해 일별 종가 기준 수집 | 실시간 틱 데이터는 불필요, 일봉으로 충분 |
| MDD 기간 | 사용자 선택 (4주, 12주, 26주, 52주) | 단기~장기 관점을 사용자가 자유롭게 선택 |
| MDD 계산 기준 | 일별 종가, 선택 기간 내 고점 대비 현재가 하락률 | 표준적인 MDD 정의 |
| 임계값 단위 | 종목별 단일 퍼센트 값 (예: -20%) | 종목마다 변동성이 다르므로 개별 설정 |
| 알림 채널 | Telegram 또는 Slack 중 택1 | MVP 범위 내 단순화 |
| 중복 알림 방지 | 동일 임계값 돌파 24시간 쿨다운 | 사용자 피로도 방지 |
| 인증 방식 | 이메일/비밀번호 자체 인증 | OAuth는 Phase 2 이후 검토 |
| 시세 수집 시점 | 한국: 장 마감 후 / 미국: 장 마감 후 | 당일 종가 확정 후 계산 |

### MDD 계산 예시

```
MDD = (현재가 - 기간 내 최고가) / 기간 내 최고가 × 100

예) 52주 최고가: 100,000원, 현재가: 75,000원
    MDD = (75,000 - 100,000) / 100,000 × 100 = -25.0%
```

## 8. 수용 기준 (Acceptance Criteria)

### AC-01: 회원가입 성공
- **Given**: 유효한 이메일과 비밀번호가 입력되었을 때
- **When**: 회원가입 요청을 하면
- **Then**: 계정이 생성되고, 비밀번호는 해시로 저장됨

### AC-02: 중복 이메일 회원가입 방지
- **Given**: 이미 가입된 이메일로
- **When**: 동일 이메일로 회원가입 요청을 하면
- **Then**: 중복 이메일 오류 메시지가 반환됨

### AC-03: 종목 등록 (유효한 심볼)
- **Given**: 로그인한 사용자가 유효한 심볼(예: 005930.KS)을 입력했을 때
- **When**: 워치리스트 종목 추가 요청을 하면
- **Then**: 종목이 등록되고, 선택 기간의 종가 데이터 수집 후 초기 MDD가 계산됨

### AC-04: 종목 등록 (무효한 심볼)
- **Given**: 존재하지 않는 심볼을 입력했을 때
- **When**: 워치리스트 종목 추가 요청을 하면
- **Then**: "유효하지 않은 종목 심볼입니다" 오류 메시지가 반환되고, DB에 저장되지 않음

### AC-05: MDD 계산 정확성
- **Given**: 선택 기간 내 최고 종가가 100,000원이고 현재 종가가 75,000원인 종목
- **When**: MDD 계산이 실행되면
- **Then**: MDD 값은 -25.00%로 계산됨

### AC-06: MDD 기간 선택
- **Given**: 사용자가 종목의 MDD 기간을 12주로 설정했을 때
- **When**: MDD가 계산되면
- **Then**: 최근 12주(약 60 거래일) 내 최고가 기준으로 MDD가 산출됨

### AC-07: 임계값 초과 알림 발송
- **Given**: 종목의 임계값이 -20%이고 Telegram이 설정된 상태에서
- **When**: MDD가 -21.5%로 갱신되면
- **Then**: Telegram으로 "종목 MDD 경보: -21.5% (임계값 -20% 초과)" 형태의 알림이 발송됨

### AC-08: 24시간 중복 알림 방지
- **Given**: 종목에서 임계값 초과 알림이 방금 발송된 상태에서
- **When**: 같은 날 MDD가 더 하락하면
- **Then**: 추가 알림이 발송되지 않음

### AC-09: 데이터 격리
- **Given**: 사용자 A가 로그인한 상태에서
- **When**: 사용자 B의 워치리스트에 접근을 시도하면
- **Then**: 접근이 거부됨

### AC-10: 알림 채널 미설정 시
- **Given**: 알림 채널을 설정하지 않은 사용자의 종목이 임계값을 초과했을 때
- **When**: MDD 갱신이 실행되면
- **Then**: 알림 발송을 시도하지 않고, 로그에 기록됨

## 10. 의존성 및 제약사항

### 외부 서비스 의존성
- **시세 API**: 서비스 정책 변경 시 시세 수집 중단 위험 → 추상화 레이어로 대체 API 교체 용이하게 설계
- **Telegram Bot API**: bot 생성 및 chat_id 사전 준비 필요 (사용자가 직접 설정)
- **Slack Incoming Webhooks**: Slack 워크스페이스 내 앱 생성 필요 (사용자가 직접 설정)

### 기술 제약
- 한국/미국 주식의 장 마감 시간이 다름 → 시세 수집 스케줄 분리 필요
- 시세 API의 Rate Limit 및 응답 형식 변경 리스크 존재

### 인프라 전제조건
- PostgreSQL 인스턴스 가동 중
- 인터넷 아웃바운드 허용 (시세 API, Telegram, Slack 호출)
